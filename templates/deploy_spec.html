必须 · deploy/Dockerfile
建议在仓库根目录下提供 deploy/Dockerfile 作为 Mini Deploy 的统一构建入口； 若未提供，平台会回退识别根目录 Dockerfile。
推荐把依赖安装层和业务代码层拆开，这样 Python 工程才能稳定复用 base stage 缓存。 
# 通用示例 / Generic example
FROM python:3.11-slim
WORKDIR /app

COPY . .
RUN pip install -r requirements.txt

CMD ["uvicorn", "main:app", "--host", "0.0.0.0", "--port", "80"]
Python · pip（requirements.txt）
Python 工程通常不需要在平台里额外选择依赖方式。
当仓库中存在 requirements.txt，平台会自动按 pip 模式处理， 平台会用 requirements.txt + Dockerfile 计算 base stage 哈希。
两者都不变时，跳过 base stage 重建，直接复用 {服务名}-base:latest。 
# Stage 1: base（依赖层 / Dependency layer）
FROM python:3.11-slim AS base
WORKDIR /app

COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

# Stage 2: app（业务层 / App layer）
FROM base AS app
WORKDIR /app

COPY . .
CMD ["uvicorn", "main:app", "--host", "0.0.0.0", "--port", "80"]
Python · uv（pyproject.toml / uv.lock）
当仓库中存在 uv.lock 时，平台会自动按 uv 模式处理， 平台会用 pyproject.toml、uv.lock 和 Dockerfile 计算 base stage 哈希。
推荐把 uv sync 放在 base stage 中，并在 app stage 只复制业务代码。 
# Stage 1: base（uv 依赖层 / uv dependency layer）
FROM python:3.11-slim AS base
WORKDIR /app

COPY --from=ghcr.io/astral-sh/uv:latest /uv /usr/local/bin/uv
COPY pyproject.toml uv.lock ./
RUN uv sync --frozen --no-dev

# Stage 2: app（业务层 / App layer）
FROM base AS app
WORKDIR /app

COPY . .
CMD ["uv", "run", "uvicorn", "main:app", "--host", "0.0.0.0", "--port", "80"]
自动识别优先级为：uv.lock → requirements.txt → pyproject.toml。 一般情况下保持默认即可，无需额外配置。
Python 工程配置建议
工程类型选择 Python，启用 base stage 缓存逻辑。
依赖方式通常无需手动指定，平台会按仓库中的 uv.lock / requirements.txt 自动识别。
Dockerfile stage请确保依赖层 stage 名称为 base，应用层 stage 建议为 app。
Java（Maven / Gradle）
工程类型设为 Java 时，平台会在临时容器内自动执行 Maven 或 Gradle 构建， 然后再使用仓库中的 Dockerfile 打包最终镜像。 
# Maven 项目 / Maven example
FROM eclipse-temurin:17-jre-alpine
WORKDIR /app
COPY target/*.jar app.jar
CMD ["java", "-jar", "app.jar"]

# Gradle 项目 / Gradle example
FROM eclipse-temurin:17-jre-alpine
WORKDIR /app
COPY build/libs/*.jar app.jar
CMD ["java", "-jar", "app.jar"]
前端（Node 构建 → 静态文件发布）
工程类型设为 前端 时，平台会在临时 Node 容器中执行构建命令， 再把 dist_dir 指向的构建产物复制到 nginx 容器或宿主机目标目录。
前端项目通常不需要 Dockerfile，但需要正确配置构建命令、代码子目录和目标路径。
可选 · deploy/hooks/ 钩子系统
平台会按顺序尝试执行以下钩子脚本；文件不存在时静默跳过，非零退出码会中止部署。
pre_build.sh构建前执行，适合生成配置或准备依赖。
post_build.sh镜像构建后执行，适合冒烟测试或推送镜像。
pre_deploy.sh部署前执行，适合迁移数据库或备份数据。
post_deploy.sh健康检查通过后执行，适合通知、预热缓存等。
钩子环境变量
IMAGE当前镜像名，如 myapp:latest
ROLLBACK_TAG回滚镜像标签，如 myapp:rollback
SERVICEdocker compose 服务名
COMPOSE_FILEdocker-compose.yml 路径
REPO_DIR临时 clone 代码目录
PROJECT_ID平台内工程 ID
部署流程
git clone --depth=1 $REPO_URL /tmp/deploy_xxx

# 备份当前镜像
docker tag $IMAGE $ROLLBACK_TAG

# 构建前钩子
bash deploy/hooks/pre_build.sh

# Python
#   pip: requirements.txt + Dockerfile 变化时重建 base
#   uv:  pyproject.toml / uv.lock + Dockerfile 变化时重建 base
#
# Java
#   Maven / Gradle 容器内构建，再使用 Dockerfile 打包镜像
#
# Frontend
#   Node 容器内执行 build_command，然后复制 dist_dir 产物

docker build -t $IMAGE -f deploy/Dockerfile .
bash deploy/hooks/post_build.sh

bash deploy/hooks/pre_deploy.sh
docker compose -f $COMPOSE_FILE up -d --no-deps $SERVICE

# 健康检查失败则自动回滚
bash deploy/hooks/post_deploy.sh
平台配置项说明
仓库地址GitHub HTTPS 地址，私有仓库可配合 Token 使用。
服务名docker-compose.yml 中的服务名。
镜像名留空时默认使用 {服务名}:latest。
Compose 路径留空时使用全局默认 compose 文件。
自重启服务用于部署 Mini Deploy 自身时的安全重启流程。
Java 版本Java 工程使用的 Maven / Gradle 基础镜像版本。
Node 版本前端工程构建使用的 Node 镜像版本。
